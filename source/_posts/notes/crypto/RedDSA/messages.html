Maybe I can slightly answer your question.

You can choose to use a curve with prime order such as P-256, however these curves are not the state of the art in terms of Elliptic curve group arithmetic performance.

AFAIK a Hisil paper from ~2008 is still the best.
Also, previously there were no complete formulas for curves such as P256, so constant time implementations were harder to achieve. There are complete formulas now, but they are still slower than the Twisted Edwards Curves group arithmetic. Not sure by how much though
19:46
I think the main downside of using Ristretto is interopability with previous protocols.

For example EdDsa is specified for twisted edwards curve which is a curve that is birationally equivalent to Curve25519. and X25519 is specified for the montgomery curve; Curve25519.

There is probably a way to make them interopable by maybe dividing by 2 because the isogeny that Ristretto uses is 2-isogenous. But subjectively judging from the design decisions on Ristretto255 repo, I think it is meant to be used with newer protocols and older protocols that explicitly ask for a group of prime order.
19:49
There is no overhead in using Ristretto versus using the underlying representation, which in the case of Ristretto255 is the birationally equivalent twisted edwards curve.

There is a small overhead in _converting_ points to "Ristretto" however I believe this is insignificant, for Decaf its roughly 1 inverse square root and 13 multiplications, so Ristretto would be a roughly +2 extra field multiplications. I guess it also depends on how you implement the inverse square root, but in general I think the performance impact should be negligible

I think with Ristretto it is better to use something like Ristretto with Schnorr. AFAICT you have to treat the Ristretto group as an opaque element, so the underlying representation may not be an Edwards curve.

In terms of transforming points, The Ristretto guidelines that Henry et al wrote up, says that you should not allow users to arbitrarily _transform_ points in and out of the Ristretto domain.

See section 5 in https://tools.ietf.org/html/draft-hdevalence-cfrg-ristretto-00
23:54
I guess the main thing is that you should not arbitrarily mix RistrettoPoints and EdwardsPoints because they are not the same mathematical objects
