https://eprint.iacr.org/2020/934.pdf CanDID: Can-Do Decentralized Identity with Legacy Compatibility, Sybil-Resistance, and Accountability

Decentralized ID is a hot topic. This paper proposes a practical, complete DID system with implementation, which other standards have failed to do so far.

The model is: committee of nodes, chosen in a permissioned way and assuming less than t<n/3 of them are faulty.
An issue I have here is that the model assumes static corruption. I think that it makes more sense in such a system for a committee to change/refresh nodes once in a while and for attacker to adaptively corrupt nodes.

At first, CanDID solves the bootstrapping problem: how to connect to existing legacy credentials. In fact, the paper is composing here a result of another (excellent!) paper: DECO (which will appear in CCS). DECO shows how to import statements from TLS communications. See fig 2.

Next come the step of issuing a master credential. This is happening using an MPC between committee members and a user. What‚Äôs nice about this MPC is that it removes dedup‚Äôs - keeping only one unique credential per one unique ID. In a followup MPC the user gets his authentication token. Because it is MPC - we get privacy all the way! The implementation of this part was done using MP-SPDZ, which is fantastic suite of MPC protocols (first time I see it being used in a paper not by Marcel).
What‚Äôs missing for me is robustness in the MPC: we assume t<n/3 so robustness is achievable, however the MPC is using n out of n protocol. Another thing missing is economic model for the nodes (but I get that‚Äôs too much to ask üôÇ )

The system enables to issue statements (called contexts) and connect them to the master credential of a user by way of zk proving a joint identifier between the statement and the master cred and have the nodes sign on the context.

Next, is an interesting ability - CanDID let the user prove (zk, circuit built with jsnark) she do not belong to a sanctions list. Since names can be misspelled, the zk actually proves something like ‚Äúedit distance from all names in the sanctions list is >0‚Äù . Similar check against a revocation list might happen from time to time and therefore the committee nodes need a tool to do it as well (remember, all credentials are private). This is done using MPC (MP-SPDZ). As it seems from the implementation, either MPC or ZK methods become extremely slow as a function of the list size.

Finally, there‚Äôs a key recovery protocol which uses the same committee to distribute the key and let a user reconstruct from them upon proving authentication from existing services (i.e. FB + Google). This concept of letting the same committee handle secrets is discussed in various other places. Using existing centralised services to get access to backup is a bit of a backdoor in the decentralised concept but there are ways to overcome it.

Imho section V (the system) is the most important and section VIII (implementation) right after.

The attacks Shlomovits and Aumasson identified would not be trivial for an attacker to carry out. They all involve a somewhat privileged position within an exchange, whether taking control altogether or controlling one portion of a distributed key. Beyond identifying individual vulnerabilities, though, the two say that the goal of the research was to call attention to how easy it is to make mistakes while implementing multiparty distributed keys for cryptocurrency exchanges. And how impactful those mistakes can be when they're in open-source libraries that can proliferate widely. Using distributed key schemes is an important defense against takeover, but the cryptography is complicated enough that the development process can't be taken lightly.
