Rushing adversary is the common model for synchronous MPC, i.e. protocol that is executed in rounds. Itâ€™s more of a network thing and a good approximation of real world. If you assume asynchronous network you donâ€™t need to assume rushing adversary. If you want to design a protocol in which you specifically assume the adversary is not the last one to answer - you are putting a strong limitation on your protocol (not saying that it is not interesting), for example - I will not use such protocol in production.
As part of the rushing adversary model, the adversary gets to decide if the honest parties get output. The way to combat this is by requiring certain security properties from the MPC protocol: fairness and guaranteed output delivery (if the adversary gets output so does the honest parties) are the most relevant. See https://eprint.iacr.org/2014/668.pdf for discussion.
