Nova: Recursive Zero-Knowledge Arguments from Folding Schemes (https://eprint.iacr.org/2021/370.pdf )

Interesting work from @srinathsetty et at !
1) (section 3) defines Folding scheme (def. 7): as the name suggests this is basically a way to fold two NP instance-witness tuples into one. It becomes non trivial when complexity is less of what required for 2x NP
2) (section 4) introduces Relaxed R1CS (RR1CS). Loosely speaking, RR1CS is R1CS with additional degrees of freedom, it is easy to show RR1CS is equivalent to R1CS which intuitively means RR1CS can replace R1CS and be as expressive
3) (section 5) combines the above two sections and constructs (construction 2) a folding scheme for committed RR1CS (CRR1CS). Note that folding is done with “committed” RR1CS, which is for zero knowledge. I am curious why R1CS can’t be folded ( I have a guess but it is not explained in the paper)
4) (section 6) [side comment: The title of the paper paves the way for the title of a follow up paper (Super Nova)]. Incremental verifiable computation (IVC) allows a prover to show that z_n = F^(n)(z_0) - essentially repeating F “n” times. In this section, IVC is constructed using CRR1CS. It is done by defining first a recursion function F’ (Fig 2) that runs F. F’ is represented as CRR1CS -> can be folded. Lemma 8 is notable as it shows that the verifier work is dominated by 2 scalar multiplications (!!). It is not clear to me how to make it a PCD but I guess it is work for supernova
5) (section 7) the missing piece. The R1CS version of Spartan is adapted to RR1CS (using spartan == no FFT, no trusted setup). This allows the prover to prove zksnark a valid NP witness for the final step of the IVC (or intermediate steps)

07:38
Adam Gagol
You could for example implement an anti-MEV DEX that accepts transactions in the following way:
1. First, tx needs to be sent to the smart contract in an encrypted form, let's say that enc(tx) is included in block K.
2. Nodes holding key shares would always deliver decrypted tx before the block K+3 (but AFTER block K is mined), so it is always included in the block K+3.
Then to frontrun this tx, you would need to include encrypted frontrunning transaction in the block K as well, so before the decrypted version of the original tx is revealed.
In general, I think you could achieve a pretty similar goal as this scheme with users having to commit to a given transaction by publishing a hash of it 3 blocks in advance (and publishing the tx later), the threshold crypto doesn't seem to be all that essential here.
